Queue Browser

In the database, jobs are organized in a hierarchy of "experiments", "passes", and "jobs", a structure that is reflected in the Queue Browser.  Experiments are essentially a "user-based" organization - they allow the user to collect a bunch of correlation activities under a common heading - for instance all correlations related to a particular study (thus the "experiment" title).  An experiment can contain any number of passes, and they don't have to be related in any particular way (its up to the user).  A pass corresponds to a single ".v2d" and ".vex" file set (where most parameters governing software correlation appear).  In the "staging area" file structure of the software correlator a unique directory is assigned to each pass (and they share a name).  This directory will contain all of the files related to the pass and its child jobs.  Jobs correspond to unique ".input" files which are generated by vex2difx in the staging area. 

You can, if you wish, completely ignore the database - it doesn't even have to exist on your software correlator.  The Queue Browser will still create jobs with the same experiment/pass/job hierarchy.

Creating a New Experiment

  • In Data Base indicates whether or not this experiment is to be contained in the DiFX data base on the data base server.  You may not have a DiFX data base or you may not be using it, in which case this item will not be checked.  If you think you are using the data base and the item is not checked the GUI was unable to communicate with the data base server, which might indicate your data base connection parameters are bad (these parameters are set in the Settings Window - see Data Base Settings).
  • Data Base ID is a unique number assigned to each experiment by the DiFX data base server software.  If In Data Base is checked, this field will initially be blank (as the data base has yet to assign a number).  If In Data Base is not checked, the GUI will attemp to assign a unique ID on its own.
